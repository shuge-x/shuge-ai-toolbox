## Context

文本摘要工具是工具箱的第一个 active 工具页面，需要作为后续工具的参考实现。技术约束：纯前端实现，不依赖外部 API，数据全在客户端处理。

## Goals / Non-Goals

**Goals:**
- 提供可用的文本摘要功能，基于规则提取关键句子
- 组件结构清晰，可作为其他工具页面的参考模板
- 遵循项目已有的 UI 模式和组件风格

**Non-Goals:**
- 不使用外部 AI API 进行摘要
- 不做流式输出或流式渲染
- 不涉及 Zustand 等全局状态管理

## Decisions

### 1. 摘要算法：基于规则的关键句提取

**考虑过：**
- TF-IDF 词频分析（需要分词库，复杂）
- 外部 API（违反纯前端约束）
- 简单规则（首句提取，太简陋）

**选择：基于位置和词频的混合策略**
- 段落首句（Topic Sentence）权重最高
- 包含高频词（除停用词外出现 2 次以上）的句子
- 文章首句通常包含核心观点
- 按摘要长度设置句子数量上限：短=3句，中=5句，长=8句

**为什么：** 无外部依赖，实现简单，对新闻类、说明类文本效果好。

### 2. 组件结构：单一文件 + 独立样式

**考虑过：**
- 拆分多个子组件（输入区、结果区）（过度设计）
- 使用 CSS Modules（增加复杂度）

**选择：单文件组件 + Tailwind classes**
- 工具页面简单，拆分反而增加维护成本
- 样式直接用 Tailwind，与项目风格一致

### 3. 状态管理：useState 本地管理

**考虑过：**
- Zustand store（过度设计，不需要共享状态）
- URL 参数持久化（暂不需要）

**选择：useState**
- 输入文本、摘要结果、长度选择都是组件内部状态
- 简单直接，符合最小复杂度原则

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 规则摘要对文学性文本效果差 | 小说、散文类无法提取有效摘要 | UI 提示「规则摘要可能不适用于文学类文本」 |
| 组件样式与 Home 页面不一致 | 视觉体验割裂 | 复用 Home 卡片的设计 tokens（border, shadow, rounded-xl） |
| 路由懒加载失败导致白屏 | 工具页面无法访问 | router/index.tsx 有 Suspense fallback 展示加载中 |
