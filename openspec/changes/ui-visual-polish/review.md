## 5. 任务粒度

**状态**：⚠️ 警告

**发现**：
- 视觉类任务（如"添加 Lucide 图标"）难以按 TDD 节奏严格拆分，测试覆盖成本高但收益有限
- 部分任务（如 Tab 滑动指示器）包含多个子步骤，需要拆分避免单 task 超过 5 分钟
- CSS 变量重建如果一次性替换所有 primary 变量，涉及文件多但逻辑简单，可合并为一个大步骤

**建议**：
- CSS 变量更新（tokens.css）→ 合并为一个任务：安装 lucide-react + 重建 primary 色板
- Home.tsx 图标 → 拆分为：Category 图标任务 + Badge 图标任务（各自独立）
- Tab 滑动指示器 → 拆分为：JS 计算逻辑任务 + CSS transition 样式任务
- Stagger 动画 → 拆分为：keyframes 定义任务 + nth-child 动画应用任务
- 测试任务可适度简化（快照测试 + 浅层渲染测试，不强求 TDD 全流程）

---

## 整体评估

**变更类型**：纯视觉实现，跨 6 个文件，涉及 1 个新依赖。

**核心风险**：
1. Tab 滑动指示器的 left/width JS 计算逻辑有侵入性，需要注意初始状态和 resize 处理
2. Stagger 动画的 CSS 写法（nth-child + calc）需要确保兼容 modern browser

**对 tasks.md 的关键建议**：
1. 以视觉任务为主线组织，而非 TDD 流程为主线
2. 将 Lucide 安装和 tokens.css 更新合并为"依赖安装 + 配色重建"任务
3. Tab 指示器单独拆出一个任务：JS ref 计算 + CSS transition
4. 为 Home.tsx 的图标渲染写快照测试（验证 icon type 正确）
5. 每个任务明确说明"改什么"，而非"测什么"
